Key Distinctions
  • LLM vs SLM: Size + scope. LLMs = broad, expensive, cloud-hosted. SLMs = narrow, efficient, can run on-device/edge.
  • GAN vs Transformer: GANs use adversarial training (generator vs discriminator). Transformers use self-attention. Both can generate content but architecturally different.
  • Statistical Learning vs Deep Learning: Statistical = interpretable, smaller data. Deep Learning = complex patterns, massive data, less interpretable.
2. Learning Types — Supervised vs Unsupervised vs Reinforcement
AspectSupervised LearningUnsupervised LearningReinforcement Learning
Training DataLabeled data (input + correct output)Unlabeled data (no correct answers)No dataset; learns from environment via trial/error
GoalPredict known outcomesDiscover hidden patterns/groupsMaximize cumulative reward

Key Distinctions
  • Supervised vs Unsupervised: The defining difference is LABELS. If the training data has correct answers attached, it is supervised.
  • Reinforcement vs Supervised: RL has no dataset at all — it learns by interacting with an environment and receiving rewards/penalties. Think "learning by doing" vs "learning from examples."
3. Model Optimization — Fine-tuning vs Pruning vs Quantization
TechniqueWhat It DoesWhen to UseTrade-off
Fine-tuningContinues training a pre-trained model on domain-specific dataAdapting a general model to your industry/task (e.g., medical LLM)Requires compute + labeled data. Risk of catastrophic forgetting.
PruningRemoves unnecessary neurons/connections (zeros out low-weight parameters)Deploying models on resource-constrained devicesSmaller model, faster inference. May lose some accuracy.
QuantizationReduces numerical precision of weights (e.g., 32-bit → 8-bit or 4-bit)Reducing memory footprint and speeding up inferenceSignificant size reduction. Small accuracy loss.
Key Distinctions
  • Fine-tuning vs Pruning/Quantization: Fine-tuning changes WHAT the model knows. Pruning and quantization change HOW EFFICIENTLY the model runs without changing its knowledge.
  • Pruning vs Quantization: Pruning removes parts of the model entirely. Quantization keeps all parts but uses less precise numbers to represent them.
4. Prompting Techniques

System vs User Prompts

Prompt TypeWho Sets ItPurposeExample
System PromptDeveloper/adminSets behavior, persona, constraints, guardrails for the entire session"You are a security analyst. Never reveal internal IP addresses."
User PromptEnd userThe actual question or task from the user"Analyze this log file for suspicious activity."

Shot Techniques

TechniqueExamples GivenBest ForPrompt Pattern
Zero-shot0 (no examples)Well-understood tasks; testing general capability"Classify this email as spam or not spam."
One-shot1 exampleWhen format/style needs to be shown once"Example: 'Free money!' → Spam. Now classify: ..."
Multi-shot (Few-shot)2+ examplesComplex tasks; establishing patterns the model should follow"Example 1: ... → Spam. Example 2: ... → Not spam. Example 3: ... → Spam. Now classify: ..."
Key Distinctions
  • System vs User prompt: System prompts are invisible to the end user and set guardrails. User prompts are the visible conversation. Attackers target system prompts via prompt injection.
  • Zero-shot vs Few-shot: Zero-shot tests the model's inherent capability. Few-shot teaches by example within the prompt. More examples = more token cost but better adherence to format.

Data Operations — Grouped by Purpose

PurposeOperationWhat It Does
Quality / PrepData CleansingRemoves errors, duplicates, inconsistencies from datasets
Data VerificationConfirms data is accurate and meets quality standards
Trust / OriginData LineageTracks data's journey: where it came from, how it was transformed
Data ProvenanceRecords the origin and ownership of data
SecurityData IntegrityEnsures data hasn't been tampered with (hashing, checksums)

Data BalancingEnsures equal representation across classes to prevent bias (oversampling minority, undersampling majority)
Key Distinctions
  • Lineage vs Provenance: Lineage = the full journey (every transformation). Provenance = specifically where it originated/who owns it. Lineage is broader.
  • Cleansing vs Verification: Cleansing fixes problems. Verification checks whether problems exist. Cleansing is the action, verification is the audit.
  • Augmentation vs Balancing: Augmentation creates more data. Balancing ensures equal class distribution. You might augment to balance, but they serve different purposes.
  • Integrity vs Verification: Integrity = was it tampered with? (security concern). Verification = is it accurate? (quality concern).
6. Key Technologies — RAG, Embeddings, Vector Storage, Watermarking

How They Connect

User Query → Embedding Model (converts text to numerical vectors) → Vector Database (searches for similar vectors) → Retrieved Context → LLM + Context = RAG Response
TechnologyWhat It DoesWhy It Matters
EmbeddingsConvert text/images into dense numerical vectors that capture semantic meaningEnable similarity search — "king" is close to "queen" in vector space
Vector Storage / Vector DBSpecialized database optimized for storing and searching embeddings (e.g., Pinecone, ChromaDB)Makes RAG fast — finds relevant documents in milliseconds
RAG (Retrieval-Augmented Generation)Retrieves relevant external data at inference time and feeds it to the LLM as contextReduces hallucinations, keeps responses current without retraining
AI WatermarkingEmbeds imperceptible markers in AI-generated content for attribution/detectionIdentifies AI-generated text/images, tracks model outputs, combats misuse
Key Distinctions
  • RAG vs Fine-tuning: RAG adds knowledge at inference time (no retraining). Fine-tuning bakes knowledge into the model weights (requires retraining). RAG = dynamic, Fine-tuning = permanent.
  • Embedding vs Vector DB: Embedding is the process/result (converting text to numbers). Vector DB is where you store and search those numbers.
7. Model Training — Epochs & Validation
ConceptDefinitionWhy It Matters
EpochOne complete pass through the entire training datasetToo few epochs = underfitting (model hasn't learned enough). Too many = overfitting (model memorized training data, fails on new data).
Model ValidationTesting model performance on a held-out dataset it wasn't trained onEnsures the model generalizes to new, unseen data and doesn't just memorize training examples.
Key Distinctions
  • Training Set vs Validation Set vs Test Set: Training = learns from it. Validation = tunes hyperparameters during training. Test = final evaluation on completely unseen data.
  • Overfitting vs Underfitting: Overfitting = great on training data, poor on new data (too many epochs, too complex). Underfitting = poor on everything (too few epochs, too simple).

Human-in-the-Loop vs Human Oversight vs Human Validation

ConceptWhenRole of HumanExample
Human-in-the-Loop (HITL)During operation, for EVERY decision or at critical pointsActively involved in the decision-making process; AI cannot proceed without human approvalA doctor reviews every AI-suggested diagnosis before it goes to the patient
Human OversightOngoing, across the lifecycleMonitors, audits, and governs AI systems at a strategic/policy level; can intervene but doesn't approve every actionAn AI governance board reviews model performance metrics quarterly
Human ValidationBefore deployment / at evaluation stageTests and confirms the model meets requirements, is safe, and performs correctly before releaseQA team runs adversarial tests on a chatbot before it goes live
Key Distinctions
  • HITL vs Oversight: HITL = operational, per-decision involvement ("approve this output"). Oversight = strategic, governance-level ("is this system working correctly overall?").
  • HITL vs Validation: HITL is ongoing during operations. Validation is a gate before deployment. HITL is continuous; validation is a checkpoint.
  • Oversight vs Validation: Oversight is broad and ongoing governance. Validation is specific testing at a point in time.
