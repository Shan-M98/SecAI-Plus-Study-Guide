Key Distinctions
  • OWASP LLM Top 10 vs ML Top 10: LLM Top 10 is specific to language models (prompt injection, hallucinations). ML Top 10 covers all ML (adversarial examples, model theft). If the question says "LLM" or "chatbot," think LLM Top 10. If it says "ML model" or "classifier," think ML Top 10.
  • MITRE ATLAS vs OWASP lists: ATLAS maps adversary TTPs (like ATT&CK for AI). OWASP lists rank vulnerabilities. ATLAS = "how attackers attack." OWASP = "what vulnerabilities exist."
  • MIT AI Risk Repository vs the others: MIT is the broadest — it includes non-security risks (bias, job displacement, misinformation). The others are security-focused.
2. Defensive Controls — Defense-in-Depth Stack

Ordered from outermost (network/access) to innermost (model behavior). Multiple layers provide defense-in-depth.

Perimeter
Endpoint Access Controls — Authentication, authorization, API keys. Who can access the AI system at all?
Perimeter
Rate Limits & Token Limits — Prevent abuse via volume. Rate limits cap requests/time. Token limits cap input/output size per request.
Input Layer
Prompt Firewalls — Inspect and filter incoming prompts for known attack patterns (injection, jailbreak attempts) before they reach the model.
Input Layer
Input Quotas & Modality Limits — Restrict what types of input are accepted (text only? images? files?) and maximum input sizes per modality.
Prompt Layer
Prompt Templates — Pre-structured prompts that constrain user input to specific fields, preventing freeform injection.
Model Layer
Guardrails — Rules enforced within or around the model: topic restrictions, output filters, content policies, safety classifiers. The last line of defense on model behavior.
Key Distinctions
  • Prompt Firewall vs Guardrails: Prompt firewalls filter INPUT before the model sees it. Guardrails constrain the model's BEHAVIOR and OUTPUT. Think: firewall = gate at the door. Guardrails = rules inside the house.
  • Prompt Templates vs Prompt Firewalls: Templates structurally prevent freeform input (the user fills in blanks). Firewalls analyze freeform input for malicious content. Templates = prevention by design. Firewalls = detection and blocking.
  • Rate Limits vs Token Limits: Rate limits = requests per time period (e.g., 100/min). Token limits = size per request (e.g., max 4096 tokens). Both prevent abuse but measure different things.
3. Attack Types — Grouped by Phase

TRAINING PHASE Attacks That Corrupt the Model Before Deployment

Data Poisoning

Injecting malicious/misleading data into the training set to corrupt model behavior.

Model Poisoning

Directly tampering with model weights or architecture (e.g., inserting backdoors into a pre-trained model).

Model Skewing

Subtly biasing training data distribution so the model favors certain outcomes (a form of data poisoning).

Transfer Learning Attacks

Exploiting vulnerabilities in a pre-trained base model that persist after fine-tuning. Attacker compromises the foundation model.

AI Supply Chain Attacks

Compromising third-party models, datasets, libraries, or frameworks that are integrated into the AI pipeline.

INFERENCE PHASE Attacks on Live Model Input

Prompt Injection

Crafting inputs that override system instructions or make the model execute unintended actions. Can be direct (user input) or indirect (hidden in retrieved data).

Jailbreaking

Bypassing the model's safety guardrails to produce restricted content. A specific type of prompt manipulation targeting safety filters.

Input Manipulation / Adversarial Examples

Subtly modifying inputs (images, text) to cause misclassification. Imperceptible to humans but fools the model.

Model DoS / Resource Exhaustion

Sending complex, resource-intensive queries to overwhelm the model's compute resources. Targets availability.

OUTPUT PHASE Problems with Model Responses

AI Hallucinations

Model confidently generates false, fabricated, or nonsensical information. Not an "attack" but a vulnerability if outputs are trusted.

Improper Output Handling

Failing to sanitize/validate model outputs before using them in downstream systems (e.g., injecting SQL via LLM output). Per OWASP LLM Top 10 (numbering varies across versions).

DATA EXTRACTION Stealing Model Knowledge or Training Data

Model Inversion

Querying the model to reconstruct sensitive training data (e.g., recovering faces used to train a facial recognition system).

Model Theft / Extraction

Querying a model extensively to replicate its functionality, creating a stolen copy. Targets intellectual property.

Membership Inference

Determining whether a specific data point was used in training. Privacy risk: "Was my medical record in the training data?"

DESIGN FLAWS Architectural / Trust Issues

Excessive Agency

Giving the AI system too many permissions, capabilities, or autonomy. It can take actions beyond what's intended (e.g., an AI assistant that can delete files).

Overreliance

Humans blindly trusting AI outputs without verification. Leads to acting on hallucinations or biased recommendations.

Key Distinctions
  • Prompt Injection vs Jailbreaking: Prompt injection overrides instructions to make the model do something unintended. Jailbreaking specifically bypasses safety filters to produce restricted content. All jailbreaking is a form of prompt manipulation, but not all prompt injection is jailbreaking.
  • Data Poisoning vs Model Poisoning: Data poisoning corrupts the training DATA. Model poisoning directly modifies the model WEIGHTS/architecture. Data poisoning is indirect; model poisoning is direct.
  • Model Inversion vs Membership Inference: Inversion reconstructs actual training data. Membership inference only determines IF a data point was used. Inversion is more damaging.
  • Model Theft vs Model Inversion: Theft steals the model's functionality (IP). Inversion steals the training data (privacy). Different targets.
  • Excessive Agency vs Overreliance: Excessive agency = the AI has too much POWER. Overreliance = the human gives too much TRUST. One is a technical flaw, the other is a human behavior problem.

Privacy Techniques

TechniqueWhat It DoesReversible?Example
AnonymizationPermanently removes all personally identifiable information. Data cannot be traced back to individuals.No (irreversible)Replace all names with random IDs, remove DOB, zip codes
RedactionPermanently removes or blacks out specific sensitive content from documents/data.No (irreversible)Blacking out SSN in a PDF before feeding to AI

Key Distinctions
  • Anonymization vs Masking: Anonymization is IRREVERSIBLE — you cannot get back to the original. Masking is REVERSIBLE — the real data exists somewhere and can be unmasked. Exam loves this difference.
  • Anonymization vs Redaction: Both are irreversible, but anonymization transforms identifiers across a dataset. Redaction removes/covers specific content in a document. Anonymization = dataset-wide. Redaction = document-level.
  • Minimization vs the others: Minimization is a COLLECTION principle (don't gather it in the first place). The others are PROCESSING techniques (handle data you already have).
  • In Use vs At Rest: "In use" encryption is the hardest to implement and most tested on the exam. If the question mentions "processing sensitive data without exposing it" or "confidential computing," the answer is encryption in use.
5. Monitoring & Logging
ControlWhat It DoesWhy It Matters for AI
Prompt MonitoringLogging and analyzing all prompts sent to the AI systemDetects prompt injection attempts, abuse patterns, policy violations. Essential for incident response.
Log SanitizationRemoving sensitive data (PII, secrets, credentials) from logs before storageAI logs may capture user inputs containing sensitive data. Unsanitized logs become a data leak vector.
Response Confidence LevelTracking and exposing the model's confidence score for each responseLow-confidence responses are more likely to be hallucinations. Can trigger human review for low-confidence outputs.
AI Cost MonitoringTracking compute costs, token usage, and resource consumptionDetects abuse (cryptomining via AI resources), budget overruns, and Model DoS attacks. Unusual cost spikes = potential attack.
Key Distinctions
  • Prompt Monitoring vs Log Sanitization: Monitoring is about ANALYZING what was sent (detection). Sanitization is about CLEANING what gets stored (privacy). You need both: monitor for attacks, but sanitize before long-term storage.
  • Response Confidence vs Hallucination: Confidence level is a METRIC. Hallucination is a FAILURE MODE. Low confidence does not always mean hallucination, and high confidence does not guarantee accuracy. But monitoring confidence helps catch hallucinations.
6. Data Classification Labels

Data classification determines what protections are required. AI systems process data at many sensitivity levels, so classification drives access controls, encryption requirements, and handling procedures.

LabelDescriptionAI Relevance
PublicNo restrictions. Open to anyone.Safe for training data, public-facing AI outputs
Internal / PrivateFor internal use only. Breach causes minor harm.Internal documents used for RAG must stay internal
ConfidentialSensitive business data. Breach causes significant harm.Customer data, proprietary models, fine-tuning datasets
Restricted / Top SecretHighest sensitivity. Breach causes severe harm.PII in training data, trade secrets in model weights, classified information
Key Distinctions
  • Classification vs Labeling: Classification is the policy/decision (this data is "confidential"). Labeling is the act of marking/tagging it so systems and people can enforce the classification.
  • Why classification matters for AI: A model trained on "restricted" data may leak that data through model inversion or memorization. The classification of training data should determine the classification of the model itself.
