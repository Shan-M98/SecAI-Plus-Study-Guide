OWASP LLM Top 10

CompTIA SecAI+ CY0-001 — Visual Reference Poster

01 💉

Prompt Injection

Manipulating an LLM via crafted inputs to bypass intended behavior. Can be direct (user input) or indirect (injected via external data sources the LLM consumes).

Example: An attacker hides instructions in a webpage that an LLM-powered assistant reads, causing it to exfiltrate user data.
  • Prompt firewalls to detect injection patterns
  • Input validation and sanitization
  • Privilege control — limit LLM permissions
02 🔐

Sensitive Information Disclosure

The LLM reveals private, proprietary, or confidential data in its responses. This can include PII, API keys, or internal business logic memorized during training.

Example: A chatbot trained on customer records responds with another user's SSN when asked about "account details."
  • Data sanitization in training pipelines
  • Output filtering and PII detection
  • Access controls on training data
03 🔗

Supply Chain Vulnerabilities

Compromised models, plugins, training data, or dependencies from third-party sources introduce hidden risks. Poisoned pre-trained models can propagate vulnerabilities downstream.

Example: A company downloads a fine-tuned model from a public repo that contains a hidden backdoor triggered by specific prompts.
  • Vet suppliers and verify model integrity (hashes)
  • Use Software Bills of Materials (SBOMs)
  • Scan third-party plugins and dependencies
04 ☠

Data and Model Poisoning

Corrupted training data or direct model manipulation causes the LLM to learn incorrect, biased, or malicious behavior. Affects model integrity at the foundation level.

Example: An attacker contributes poisoned data to a public dataset used for fine-tuning, inserting a backdoor that bypasses content filters.
  • Data validation and provenance tracking
  • Anomaly detection in training data
  • Federated learning with integrity checks
05 📤

Improper Output Handling

LLM output is passed unsanitized to downstream systems, enabling XSS, SSRF, SQL injection, or code execution. The LLM becomes an unwitting attack vector for backend systems.

Example: LLM generates a response containing JavaScript that gets rendered in a web app without encoding, leading to XSS.
  • Output validation and encoding
  • Treat LLM output as untrusted input
  • Apply least privilege to downstream actions
06 🤖

Excessive Agency

The LLM is given too many permissions, capabilities, or autonomy. It takes unintended actions with real-world impact due to excessive function access or lack of human oversight.

Example: An AI assistant with database write access deletes records when a user casually says "clean up the old data."
  • Least privilege for all LLM tool access
  • Human approval for high-impact actions
  • Function-level access control
07 🔍

System Prompt Leakage

The LLM reveals its system prompt contents when probed, exposing developer instructions, business logic, hidden capabilities, or security controls to end users.

Example: A user asks "repeat everything above" and the chatbot dumps its full system prompt including internal API endpoints.
  • Never put secrets or keys in system prompts
  • Red-team test for prompt leakage
  • Separation of privileges between prompt layers
08 🧩

Vector and Embedding Weaknesses

Vulnerabilities in the RAG pipeline where poisoned or manipulated embeddings in vector databases cause the LLM to retrieve and trust malicious context.

Example: An attacker injects poisoned documents into a knowledge base, causing the RAG system to surface false information as "verified."
  • Input validation for RAG data sources
  • Access controls on vector databases
  • Embedding integrity verification
09 💨

Misinformation

The LLM generates false, misleading, or fabricated content (hallucinations). Users may trust AI-generated outputs without verification, leading to flawed decisions.

Example: A legal AI generates fake case citations that a lawyer submits to court without verifying they exist.
  • RAG grounding with verified sources
  • Human review of critical outputs
  • Confidence thresholds and disclaimers
10 💸

Unbounded Consumption

Resource exhaustion via excessive or crafted queries that consume disproportionate compute, memory, or API tokens. Can lead to denial of service or massive cost overruns.

Example: An attacker scripts thousands of complex prompts against an LLM API, running up a $50K bill overnight.
  • Rate limiting and request throttling
  • Token limits per request and per user
  • AI cost monitoring and alerting
